# Architecture Rules (Invariants)

These rules are non-negotiable. Every agent must follow them. Violating these rules is a merge-blocking defect.

## Data Layer

1. All primary keys are UUIDs generated by the database (`gen_random_uuid()`).
2. All monetary values are integers in paisa. No floats, no decimals for money.
3. All tables have `created_at TIMESTAMPTZ DEFAULT NOW()` and `updated_at TIMESTAMPTZ DEFAULT NOW()`.
4. All business entity tables have `deleted_at TIMESTAMPTZ` for soft delete.
5. All queries on business entities filter `WHERE deleted_at IS NULL` unless explicitly querying deleted records.
6. Foreign keys always have explicit `ON DELETE` behavior defined.
7. Indexes exist on every foreign key column and every column used in WHERE/JOIN/ORDER BY.
8. Cross-table mutations use database transactions. No partial writes.

## Backend Layer

9. Controllers contain zero business logic. They validate input, call service, format response.
10. Services contain ALL business logic. They are independently testable without HTTP.
11. Repositories contain database queries only. No conditionals, no calculations, no validation.
12. All inputs are validated with Pydantic schemas before reaching the service layer.
13. All errors use typed exception classes (NotFoundError, ValidationError, etc.). No bare strings.
14. All API responses use the standard format: `{success, data, message}` or `{success, error, code}`.
15. Authentication is enforced by middleware on all routes except health check and login.
16. Authorization is enforced by role guard decorators on mutation endpoints.

## Frontend Layer

17. All server state managed by TanStack Query. No manual fetch + useState for API data.
18. All forms use React Hook Form + Zod validation.
19. All pages handle three states: loading skeleton, error display, empty state.
20. No direct API calls from components. All calls go through service files.

## Process

21. Every journal entry balances: sum(debit_amount) == sum(credit_amount). Enforced at service level and verified by tests.
22. Every stock change creates a stock_movement record. No silent stock modifications.
23. Sale recording is atomic: sale record + sale items + stock deductions + stock movements + journal entry — all in one transaction. If any step fails, entire sale rolls back.
24. PO numbers and sale numbers are auto-generated and immutable after creation.
25. No module may import from another module's repository directly. Cross-module access goes through services.
